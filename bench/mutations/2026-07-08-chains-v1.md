First run of `defn-bench --chains-only` — multi-op / cross-file cases
that the single-op mutation bench doesn't test. Shipped in commits
`01841f2` (harness), plus the projection-op path improvements from

## What actually happened

### triple-op-one-def

**Files-mode:** one Read (25,254 in tok) + one Edit that made all
three changes in a single string-replace. Two calls. Passed.

**Defn-mode:** 6 calls, 140 output tokens. The agent DID NOT use the
`code(op:"apply", operations:[...])` batch even though the mutation
bench preamble explicitly teaches it (commit `f3c5854`). It made
separate `code(op:...)` calls for each of the 3 projection ops, plus
~3 overhead. Passed correctness, but paid the per-op overhead 3×.

**Interpretation:** the preamble mention isn't enough of a nudge.
Agents fall into "one op at a time" pattern by default. Getting them
to batch may need either (a) more prominent preamble placement, (b) an
error message when 3 ops arrive in quick succession suggesting apply,
or (c) making apply the DEFAULT shape (e.g., every op returns a hint
"if you have more changes, put them in this apply payload").

### rename-across-callers (FAILURE)

**Files-mode:** 6 calls, 49,482 input tokens. Read all 3 files, Edit
all 3 files. Slow but correct.

**Defn-mode:** 4 calls, 25,990 input tokens. Defn used ~half the
input tokens because it skipped Reading the files. But it failed the
post-condition — the rename didn't propagate to all callers.

**Root cause unknown.** The raw stream-json wasn't preserved by the
harness (v1 chains-bench limitation). Two hypotheses:

1. Agent used the wrong op — e.g., renamed the def but didn't trigger
   caller updates. `code(op:"rename")` is supposed to update callers
   automatically via astRename on each GetCallers result. Possible
   the agent used something else (edit-fragment on core.go, missing
   the callers).

2. `code(op:"rename")` has a real bug where astRename doesn't touch
   the callers on this fixture shape. Would need repro on a small
   test to confirm.

**Do not cite the 30% input-token reduction on this case as a defn
win until the correctness failure is understood.** Currently defn is
30% faster at producing the wrong answer.

## What we CAN claim from this run

- **Nothing quantitative.** 1/2 correctness in defn-mode means the
  bench is not yet a clean measurement.
- **Qualitatively:** cross-file operations are where defn's Read-tax
  avoidance shows up (25,990 vs 49,482 = ~half). If we can fix the
  correctness bug, this is the workload class where defn genuinely
  wins on input tokens.

## What we CANNOT claim

- "30% input token reduction" — the 30% is on a failed run. Marketing
  claim would be dishonest.
- "Multi-op batching works" — the agent didn't use it. That's a
  preamble/prompt/API-ergonomic problem, not a technical win.
- "Defn wins on chains" — 1/2 correctness is worse than files-mode's
  2/2. Currently defn LOSES on the correctness metric.

## Followups

1. **Save raw stream-json when a case fails.** Small harness change:
   write `rawOutput` to `bench/mutations/2026-07-08-chains-fail-<name>.txt`
   for any case that flunks the post-condition. Same shape as
   `scratchpad/addimport-defn.jsonl` from earlier debug runs.

2. **Debug rename-across-callers.** With a preserved trace: check
   which op the agent actually called. If `code(op:"rename")`, it's a
   defn bug — write a failing test and fix. If something else, it's a
   preamble problem — the agent doesn't know rename cascades.

3. **Get the agent to actually use `apply`.** Options above (prominent
   preamble, error suggestion, default response hint). Cheapest first.

4. **Add a big-file rename fixture.** Current rename-across-callers
   uses 3 tiny files. A 5-file version with each file being 100+
   lines would let the Read-tax delta become dramatic (a real "wins
   by orders of magnitude" case, not a 30% one).
