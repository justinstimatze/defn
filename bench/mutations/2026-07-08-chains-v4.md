## The bench-harness path still fails — root cause not yet nailed

Failure dump preserved DB state at post-condition check time:

    id | name        | source_file
    ---|-------------|------------
    1  | RunA        | caller_a.go
    2  | RunB        | caller_b.go
    3  | ProcessData | core.go       ← should be Handle
    4  | Handle      | core.go       ← should not exist

Combined with the message `defn: salvaged empty journal.idx (renamed
to journal.idx.empty.bak); Dolt will rebuild from journal` seen in
the post-run `defn query` output, this points at either:

1. **Dolt journal race**: the rename's UPDATE landed in a working set
   that got corrupted / lost when the serve exited before autoCommit
   ran, then reconstruction from journal re-inserted the pre-rename
   state PLUS the added Handle from ingestAndResolve's re-ingest.

2. **ingestAndResolve interference**: `newMCPServer` spawns
   `go s.ingestAndResolve()` at startup which loads packages
   asynchronously. If the rename op fires before that goroutine
   completes, the cached AST from LoadAll (which still has
   ProcessData) gets ingested AFTER the rename UPDATE, inserting a
   new ProcessData row while leaving the rename's Handle row in place.

Both hypotheses fit the observed data. Distinguishing requires either
`fmt.Println` instrumentation in the serve or a synthetic reproduction
of the bench's exact timing.

## Followups

1. Add a synchronous variant of `ingestAndResolve` at MCP server
   startup, gated by an env var. If the bench passes with sync-mode
   set, we know it's the `go s.ingestAndResolve()` race.
2. Explicit `s.db.Commit()` at the end of `handleRename` (belt-and-
   suspenders vs journal corruption), then re-run.
3. Longer-term: make write ops wait for `s.ready.Load()` before
   proceeding. Belt-and-suspenders and semantically cleaner.
