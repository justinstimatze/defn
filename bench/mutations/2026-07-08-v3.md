## What the numbers actually say

### 1. Headless `claude -p` has a ~25k input-token floor

Every case pays ~25-28k input tokens as preamble + tool schema + system
context, whether the fixture is 10 lines or 400. Delta effects from
fixture size get partly absorbed by that floor. The mutation bench is
therefore a *bad* place to shop for defn's read-tax win — its cheapest
per-case Read cost is fixed too high for the delta to bite hard.

### 2. big-replace-slice is where the read-tax bite finally shows

files-mode input tokens jumped 25,790 → 37,960 (a 47% increase) from
small to big replace-slice, all attributable to Reading the 400-line
fixture. defn-mode's input stayed near-flat (28,682): the projection op
doesn't need the file bytes, just the def graph. On this case defn wins
24% on input tokens.

### 3. big-rename-param is where tool-call count flips

files-mode did 5 calls (Edit + retry, hitting each `data` reference in
turn). defn-mode did 4. This is the FIRST case in any bench run where
files-mode had more tool calls — because `rename-param` is exactly the
op designed for "one identifier used N times across a scope."

### 4. Output tokens: consistent 25% win

Across every case, the agent using defn writes less. It reasons less,
explains less, produces terser final messages. This is the smallest
numeric effect but the most robust — it's every single case, not just
the ones with big fixtures.

### 5. Cached-token inflation

defn-mode's cache reads (~260-400k per case, vs ~80-100k for
files-mode) suggest the MCP tool schemas are being re-cached each turn.
That's a cost we're not converting to dollars in the harness but it's
real. A future bench should capture actual API cost, not just token
counts.

## What we should NOT claim from this

- "defn saves 46% on tool calls" — that was the old cmd/defn-bench
  read-question bench. Different regime. This bench doesn't touch it.
- "defn saves N% on tokens" without the qualifier that headless mode
  has a fixed floor. Real interactive-mode CLI use amortizes that
  floor over many turns, so real-user token savings look different.
- "The fixtures are big enough" — 400 lines is still small. Whether
  defn's advantage grows super-linearly on ≥2000-line files is an
  open question the current harness doesn't answer.

## What we CAN claim

- **On tiny fixtures (<50 lines), the projection ops LOSE on tool
  calls and are approximately even on tokens.** Use Read+Edit.
- **On mid-size fixtures with wide-scope edits (rename touching N
  places), defn's ops LEVEL or WIN on tool calls, WIN on output
  tokens, sometimes WIN on input tokens.**
- **Every op works correctly.** No mode ever failed a post-condition
  in this run (16/16).

## Follow-ups the next session might want

1. **Cost, not tokens.** Convert to $ using the actual per-token
   rates; input-token cache reads are cheap and inflate the "tokens
   used" number without hitting the wallet.
2. **Larger fixtures.** 2000-line real files where the Read cost
   would dwarf the ~25k floor.
3. **Multi-op chains.** "Rename param, then wrap the whole body in a
   defer, then insert a precondition" — three consecutive edits.
   Files-mode pays Read three times; defn pays zero. This is where
   the DB-as-state architecture should truly shine.
4. **Ergonomics on all ops** — the add-import file-inference fix
   (commit `5d7ad27`) landed only for add-import. Similar
   optional-field fixes on the others might close more of the
   headless variance gap.
